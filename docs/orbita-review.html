1.3 Открытые методологические вопросы
  1. Тех.решения (SDK) — нужен отдельный Gate-путь (W4-035). Укороченный: Заявка → Проектирование → Реализация → Приёмка
  2. B2B/B2G специфика — нет отдельных веток стадий по сегментам
  3. Kill-условия — описаны в методологии, но нет конфигуратора в UI
  4. DoR/DoD — стандарт описан в rwb-product-os-v3, но нет интерактивного конструктора
  5. Ретроспектива — нет шаблона в платформе

2.1 Архитектура
Текущее состояние
  • Stack
    Vanilla HTML / CSS / JS, GitHub Pages
  • Хранение
    localStorage (ограничение: single-user, нет синхронизации)
  • Auth
    OTP-имитация (нет реальной отправки кода), localStorage сессии
  • Шрифты
    Self-hosted Inter (0 CDN-зависимостей)
  • Деплой
    GitHub Pages → нужна миграция на корп. Git
2.2 Технический долг
Проблема Критичность Решение Усилие
localStorage = single-user Высокая Backend API (JSON API или Supabase) 2-3 недели
Auth = имитация OTP Высокая SSO / WB ID интеграция 1-2 недели
Нет синхронизации между пользователями Высокая WebSocket или polling + backend 2-3 недели
Нет бэкапов данных Средняя Backend + scheduled exports 1 неделя
Нет CI/CD Средняя GitLab CI на корп. Git 2-3 дня
XSS-защита базовая Низкая CSP headers + escaping audit 1 неделя
Нет тестов Низкая E2E тесты (Playwright) 2 недели
2.3 План миграции на корп. инфраструктуру
  1. Создать репозиторий на корп. GitLab
  2. Настроить CI/CD (static deploy на внутренний сервер)
  3. Перенести данные из localStorage в backend API
  4. Подключить SSO / WB ID
  5. Настроить мониторинг и бэкапы
