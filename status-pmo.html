Главное
Проектный офис — связь бизнеса и продуктовых команд. Единая картина портфеля, контроль сроков, приоритизация. Без ПО статус продукта — только у его менеджера. За три месяца ПО собрал картину по портфелю, предложил правила работы и прототип инструмента. Дальше не сдвинется без трёх решений:
1. Дать мандат ПО
Подчинение, зона ответственности, связь с PM
2. Утвердить подход
Как управляем портфелем: стадии, приоритеты, инструмент
3. Выделить людей
Два менеджера в ПО + разработчики для платформы (через CTO)
1. Обзор проектов департамента
Что есть. Пять ИТ-продуктов в контуре Проектного офиса, ещё пять перешли в другие департаменты. Из пяти продуктов один в проде, четыре в разработке. У трёх нет целей на год. Следующий шаг — спланировать 2 квартал по методологии.

Решения Требует решения
Критичные — без них не двигаемся
Место ПО в структуре — За что отвечает, кому отчитывается, что может требовать. Решается на встрече.
Утверждение методологии — Правила готовы. Без ОК команды им не следуют. Альтернатива: пилот на 2 продуктах без утверждения — но без масштабирования.
PM отчитываются в ПО — Без этого статус каждого продукта — только у его PM. Альтернатива: ПО собирает данные вручную — медленнее и неполно.
Ресурсные — можно запускать поэтапно
Два менеджера в ПОДозагрузка существующих сотрудников, 2-4 ч/нед. Минимальный вариант: один менеджер — покроет 3 продукта из 5.
Разработчики через CTO1 инфраструктурщик + 1 серверный разработчик, ~2 мес. на миграцию. Альтернатива: оставить внешний хостинг на квартал, но без корп. ИБ.
Доступ к GitLab RWBКод готов. Запрос через CTO.
