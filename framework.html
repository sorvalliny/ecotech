Назначение команды на инициативу

Процесс назначения

  1. Заявка подана — бриф попадает в реестр идей
  2. PMO Lead скринит — проверяет соответствие стратегии, дубли
  3. PMO-менеджер подбирает команду:
    • PM (владелец) — из пула доступных PM или назначение нового
    • Tech Lead — по стеку продукта, согласование с CTO
    • Дизайнер — по загрузке, из общего пула
    • Аналитик — при наличии, или PM совмещает
  4. PMO Lead утверждает — команда зафиксирована в карточке продукта
  5. PM получает уведомление — и доступ к workspace
SLA: Назначение команды — до 5 рабочих дней после Gate 0.
Инструмент: Реестр команд + Администрирование
Интеграция с внешними трекерами

Как связать ОРБИТА с YouTrack / Jira / Linear

ОРБИТА = стратегический уровень (цели, инициативы, Gate). Трекер задач = операционный уровень (эпики, спринты, задачи). Связь через URL.

ОРБИТА (стратегия)
Инициатива → RICE → Квартал → Цель H1/H2
Трекер (операции)
Эпик → Спринт → Задачи → Исполнители

Процесс связывания

  1. PM создаёт инициативу в ОРБИТА (workspace → инициативы)
  2. PM создаёт эпик в YouTrack/Jira по этой инициативе
  3. PM вставляет URL эпика в поле "Внешний трекер" инициативы
  4. Команда работает в трекере, PM обновляет статус в ОРБИТА
  5. При закрытии эпика → PM закрывает инициативу в ОРБИТА
Правило: Каждая инициатива в статусе "В работе" должна иметь ссылку на внешний трекер. PMO проверяет еженедельно.
Комитеты и Gate Review

Каждый Gate -- решение о переходе на следующую стадию (орбиту). Без прохождения Gate продукт не двигается вперёд.

Go

Все критерии выполнены. Продукт переходит на следующую стадию.

No-Go

Критерии не выполнены. План доработки 5 дней. Макс. 2 попытки.

Revise

Частичное одобрение с условиями. PM устраняет без повторного комитета.

Структура Gate-комитетов

  • Gate 0 -- PMO Lead, 5-10 дней, рассмотрение заявки
  • Gate 1 -- PMO Lead + Lead, 3 дня, комитет 30 мин
  • Gate 2 -- PMO Lead + CTO, 5 дней, комитет 45 мин
  • Gate 3 -- PMO Lead + Директор, ежеквартально, ревью 60 мин
  • Gate 4 -- PMO Lead + CTO, стратегический ревью
  • Gate 5 -- PMO Lead + CEO BU, ежегодно (QBR Q4)

Процесс при No-Go

  1. Фиксация причин -- протокол с конкретными замечаниями
  2. План доработки -- PM формирует за 5 рабочих дней
  3. Повторный Gate -- через 2-4 недели
  4. После 2-го No-Go -- эскалация PMO Lead (доработка / возврат / архив)

Процесс эскалации

Когда эскалировать:

  • Gate заблокирован > 2 недель без решения
  • Критический блокер без владельца
  • Конфликт приоритетов между продуктами
  • Второй No-Go на одном Gate

Канал: Уведомление в ОРБИТА → PMO Lead → решение в течение 3 рабочих дней

Форма: Продукт, стадия, блокер, предложенное решение, срочность (критическая / высокая)

Лог: Все эскалации фиксируются в карточке продукта (таб "Лог")

Уведомления

После каждого Gate решения система уведомляет:

  • Go: PM, команда, заказчик — продукт переходит на следующую стадию
  • No-Go: PM — план доработки, PMO — контроль сроков
  • Revise: PM — условия для устранения

Канал: уведомления в ОРБИТА (колокольчик в навигации)

Портфельный отчёт (ежемесячный)

PMO Lead готовит для руководства:

  1. Статус портфеля: продуктов всего / активных / на Gate / заблокированных
  2. Светофор по продуктам: зелёный / жёлтый / красный + комментарий PM
  3. Ближайшие Gate: какие продукты готовятся к переходу
  4. Ключевые решения: что требует внимания руководства

Статусы здоровья продукта

● On-track
Метрики в норме. Gate Review по плану. Нет блокеров.
● At-risk
Метрики ниже порога или цели не определены. Требует внимания PMO.
● Blocked
Критический блокер. Продукт не двигается. Эскалация обязательна.

Когда менять статус

  • PM обновляет статус еженедельно в карточке продукта (workspace)
  • At-risk — если фокусная метрика ниже порога Go более 2 недель
  • Blocked — если нет прогресса более 2 недель и есть внешняя зависимость
  • PMO Lead проверяет статусы на еженедельном синке
Онбординг PM

Чеклист для нового PM, назначенного на продукт.

День 1

  • ☐ Получить доступ к ОРБИТА (email от PMO)
  • ☐ Открыть Каталог → найти свой продукт → перейти в Workspace
  • ☐ Прочитать паспорт продукта: стадия, статус, описание, KPI
  • ☐ Встреча с PMO Lead — контекст, ожидания, ближайший Gate
  • ☐ Изучить эту страницу: Gate Review + Роли

Неделя 1

  • ☐ Познакомиться с командой (если назначена) → Реестр команд
  • ☐ Изучить инициативы продукта → Реестр инициатив
  • ☐ Подключить внешний трекер (YouTrack/Jira) → см. Трекеры
  • ☐ Поставить цели на текущий период → workspace → Цели
  • ☐ Пройти первый синк с PMO (статус, блокеры, план)

Месяц 1

  • ☐ Провести ≥ 3 CustDev-интервью (если стадия Разведка)
  • ☐ Подготовить артефакты для ближайшего Gate → Pre-Gate чеклист
  • ☐ Обновить RICE-оценки всех инициатив → RICE-оценка
  • ☐ Первая ретроспектива (Start/Stop/Continue)
Наставник: PMO-менеджер вашего департамента. Все вопросы — через ОРБИТА (уведомления) или на еженедельном синке.
Подготовка к Gate Review (Pre-Gate)

Начинайте готовиться за 2 недели до запланированной даты Gate.

Чеклист Pre-Gate

  1. Проверить артефакты — все обязательные документы для текущего Gate готовы? (см. Gate-чеклист)
  2. Обновить метрики — фокусные метрики стадии актуальны? (см. Метрики по стадиям)
  3. Pre-Gate синк с PMO Lead — за 7-10 дней до Gate. Обсудить:
    • Готовность артефактов (что готово, что нет)
    • Проблемные метрики (ниже порога? почему?)
    • Блокеры и риски
    • Состав участников комитета
  4. Подготовить презентацию — 5-7 слайдов:
    • Текущая стадия и что сделано
    • Метрики (факт vs порог)
    • Топ-3 инициативы на следующий квартал
    • Риски и запрос решений
  5. Календарь — убедиться что Gate Review есть в Календаре и участники приглашены
  6. Заполнить протокол — черновик протокола Gate Review (шаблон в Gate Review)
Если не готов к Gate: сообщите PMO Lead за 5 дней. Перенос Gate — нормально. Провал Gate без подготовки — нет.
RICE-оценка инициатив

RICE — метод приоритизации инициатив. Каждая инициатива в Реестре оценивается по 4 параметрам.

Формула

RICE = (Reach × Impact × Confidence) ÷ Effort
Параметр Шкала Что оценивает Пример (низкий) Пример (высокий)
Reach1-10Сколько пользователей затронет за квартал1 = <100 юзеров10 = 100K+ юзеров
Impact1-10Насколько сильно повлияет на метрику1 = минимально10 = кратный рост
Confidence1-10Уверенность в оценках (есть ли данные?)1 = гипотеза10 = подтверждено A/B
Effort1-10Трудозатраты команды (человеко-недели)1 = <1 недели10 = 3+ месяца

Интерпретация

  • RICE ≥ 60 — высокий приоритет, брать в текущий квартал
  • RICE 20-59 — средний, планировать на следующий квартал
  • RICE < 20 — низкий, оставить в бэклоге или отклонить

Пример

Инициатива: «Воронка B2C — оптимизация чекаута»
Reach: 8 (50K+ юзеров проходят чекаут/мес)
Impact: 7 (конверсия +15-20%)
Confidence: 9 (данные A/B теста)
Effort: 4 (2-3 недели разработки)
RICE = (8 × 7 × 9) ÷ 4 = 126 → высокий приоритет
Ретроспектива

Ретроспектива — обязательный процесс для всех типов продуктов. Без неё портфель не учится.

Когда проводить

  • Продукт / Edu: после каждого Gate Review (независимо от решения)
  • Тех.решение: после закрытия ключевой инициативы
  • Проект: после завершения (обязательно)
  • Квартальная: в конце каждого квартала — портфельная ретро PMO

Формат: Start / Stop / Continue

Start
Что начать делать? Новые практики, инструменты, процессы.
Stop
Что прекратить? Лишние встречи, дублирование, неэффективные процессы.
Continue
Что продолжать? Работающие практики, полезные ритуалы.

Кто участвует

  • Продуктовая ретро: PM + команда (TL, дизайнер, аналитик)
  • Портфельная ретро: PMO Lead + все PM + Lead

Результат

  • 3-5 Action Items с владельцами и сроками
  • Фиксация в workspace продукта (таб «Лог»)
  • Портфельные выводы → в квартальный отчёт PMO Lead
Правило: Ретро без Action Items = ретро не было. Минимум 1 конкретное действие.
