Что такое ОРБИТА

ОРБИТА -- система управления продуктовым портфелем для экосистемы RWB.

Помогает:

  • Запускать цифровые решения по единому стандарту
  • Видеть статус всех продуктов в реальном времени
  • Планировать на квартал и полугодие
  • Проходить Gate Review для перехода между стадиями
Принцип: каждый продукт проходит 7 орбит зрелости -- от заявки до масштабирования.
Каждая орбита = набор процессов, артефактов и критериев перехода.

Сущности ОРБИТА
Продукт
Цифровое решение в экосистеме RWB. Имеет стадию зрелости (O*0 -- O*5), команду, цели и инициативы.
Примеры: маркетплейс, сервис обучения, платформа знакомств.
Цель
Измеримый результат на полугодие (H1/H2). Привязана к OKR компании.
Инициативы декомпозируют цели на конкретные задачи. Key Results = метрики достижения.
Инициатива
Единица работы внутри продукта. Типы:
  • Бизнес-инициативы (новый рынок, монетизация, партнёрства)
  • Продуктовые (фичи, UX-улучшения, метрики)
  • Технические (инфраструктура, интеграции, рефакторинг)
Каждая инициатива имеет: RICE-оценку, статус, привязку к цели, ответственного PM.

Заявка — Предстарт
Заказчик формулирует проблему · PMO оценивает · Команда собирается
до 5 рабочих дней
Процесс
  1. Заказчик заполняет Бриф инициативы: проблема, аудитория, ожидаемый эффект, связь с OKR
  2. PMO проверяет полноту брифа, ищет дубли в реестре инициатив
  3. Скоринг по матрице приоритетов (RICE / T-shirt size)
  4. Поиск и согласование команды: PM, TL, дизайнер, DA
  5. Калибровка на Продуктовом синке или Проектном комитете
  6. Gate 0 — решение PMO за 5 рабочих дней
Артефакты
  • ·Бриф инициативы — заполненный заказчиком
  • ·Матрица скоринга — RICE оценка PMO
  • ·Карточка в реестре — статус, команда, ID
  • ·Решение Gate 0 — Go / No-Go / В бэклог
Инструменты
  • →Форма брифа (шаблон PMO)
  • →Реестр инициатив · ПИ Реестр
  • →Матрица RICE / ICE для скоринга
  • →Notion / Confluence для карточки
  • →WB Messenger / Почта для коммуникации
Gate 0 → Орбита 1 · Критерии
  • Проблема описана одним предложением и понятна заказчику
  • Целевая аудитория определена (сегмент + объём)
  • Связь с OKR компании или департамента подтверждена
  • Нет дублей в реестре инициатив
  • Команда найдена или путь к поиску согласован

Разведка — Исследование
Подтвердить Problem-Solution Fit · Определить MVP scope · ADR
3–6 недель
Процесс
  1. Kick-off: команда фиксирует цели Разведки и план исследования
  2. Qualitative: custdev-интервью (мин. 7–10 носителей проблемы)
  3. Quantitative: аналитика, бенчмарки, сегменты данных
  4. Синтез: Jobs-to-be-Done, карта болей, концепция решения
  5. Технический скоупинг: TL проводит оценку, пишет ADR при необходимости
  6. Discovery Report + прототип/story map → Gate 1
Артефакты
  • ·Интервью-гид — скрипт для Custdev
  • ·Синтез интервью — Jobs-to-be-Done, боли, цитаты
  • ·Discovery Report — находки + рекомендация Go/No-Go
  • ·ADR — Architecture Decision Record
  • ·Прототип / концепт — Figma, lo-fi или story map
  • ·Скоупинг MVP — что входит / что нет (In/Out of scope)
Инструменты
  • →Miro — синтез, story map, карта гипотез
  • →Figma — прототип и lo-fi концепт
  • →Notion / Confluence — Discovery Report
  • →SQL / Metabase — продуктовая аналитика
  • →Calendly — рекрутинг респондентов
  • →ПИ Реестр — статус Разведки для PMO
Gate 1 → Орбита 2 · Критерии
  • Problem-Solution Fit подтверждён данными (мин. 7 интервью)
  • Техническая реализуемость подтверждена (ADR написан)
  • Scope и сроки MVP зафиксированы и согласованы с TL
  • Бюджет предварительно согласован с Финансами
  • Discovery Report утверждён PM и TL

Прототип — Разработка
Разработать MVP и проверить с реальными пользователями
6–12 недель
Процесс
  1. PRD — Product Requirements Document: требования, user stories, DoD
  2. Sprint 0: инфраструктура, архитектура, CI/CD, настройка среды
  3. Итеративная разработка (спринты 1–N): DoR → Dev → Review → DoD
  4. UAT / Staging: тест с пилотными пользователями
  5. Soft launch: ограниченный выкат (beta-группа или % трафика)
  6. Gate 2 — метрики запуска + фидбек → решение о full launch
Артефакты
  • ·PRD / Спецификация — перед стартом разработки
  • ·Sprint Goal + Backlog — каждый спринт
  • ·DoR / DoD чеклист — критерии готовности задач
  • ·Release Notes — при каждом релизе
  • ·Dashboard метрик — baseline после запуска
  • ·Gate 2 Review — презентация для Комитета
Инструменты
  • →Jira / Linear — task management и спринты
  • →GitHub / GitLab — код, PR, CI/CD
  • →Figma — дизайн и developer handoff
  • →Sentry / DataDog — мониторинг и ошибки
  • →Metabase / Amplitude — аналитика запуска
  • →ПИ Реестр — статус для PMO и стейкхолдеров
Gate 2 → Орбита 3 · Критерии
  • MVP запущен в production (stable, 0 критических P0/P1 багов)
  • Базовые метрики зафиксированы (есть baseline для сравнения)
  • Первичный фидбек получен (NPS / опрос / интервью с первыми юзерами)
  • Unit economics рассчитаны предварительно (CAC, LTV)
  • Команда сформирована для стадии Роста

Разгон — Growth
GTM план, коммуникации, поддержка · Активный рост метрик · PMF
3–6 месяцев
Процесс
  1. Установка baseline: retention, DAU/MAU, NPS, funnel конверсии
  2. Анализ воронки: где теряем пользователей, точки оттока
  3. Формирование Growth Backlog: гипотезы роста (JTBD + данные)
  4. A/B тестирование: 2–3 параллельных эксперимента
  5. Итерации: Kill / Scale / Iterate по результатам тестов
  6. Quarterly Review: метрики + roadmap → Gate 3
Артефакты
  • ·Growth Backlog — приоритизированные гипотезы
  • ·A/B тест-план — гипотеза, метрика, sample size, срок
  • ·Metrics Dashboard — живой дашборд для стейкхолдеров
  • ·Quarterly Metrics Review — ежеквартальный отчёт
  • ·Roadmap развития — на следующий период
Инструменты
  • →Amplitude / Mixpanel — product analytics
  • →PostHog / VWO — A/B тесты и feature flags
  • →Looker / Metabase — дашборды и отчёты
  • →Notion — Growth Backlog, гипотезы
  • →ПИ Реестр — бэклог и статус в портфеле
  • →ПИ Стратегия — связь с OKR
Gate 3 → Орбита 4 · Критерии
  • PMF достигнут: Retention D30 ≥ бенчмарк, NPS ≥ целевого
  • Unit economics позитивны: CAC < LTV, Payback Period < 12 мес
  • Маркетинговые каналы определены, CAC измерен по каждому
  • Команда готова к масштабированию (найм, процессы)
  • Roadmap экосистемного развития согласован с CPO

Внедрение
Пилотный запуск фреймворка
Пошаговый план внедрения: от выбора пилотной инициативы до масштабирования на весь портфель продуктов.
Цель пилота — пройти с одной инициативой через орбиты О·0 → О·1 → Gate 1, получить обратную связь команды и стейкхолдеров, выявить точки адаптации фреймворка под контекст компании. После успешного пилота — масштабирование на весь портфель.
1
Выбор пилотной инициативы и команды
Выбираем 1–2 инициативы из бэклога с наибольшим потенциалом: чёткая проблема, мотивированная команда, умеренная сложность. Идеально — инициатива в фазе О·0 (ещё не стартовала в Разведке).
Критерий: мотивированный PM + TL Срок выбора: 1 неделя Ответственный: CPO + PMO
2
Установочная сессия (Kick-off)
2-часовая сессия с пилотной командой: знакомство с фреймворком, разбор каждой стадии на примере их инициативы, распределение ролей, фиксация вопросов и точек риска. Команда заполняет Бриф инициативы.
Артефакт: Бриф инициативы Формат: воркшоп 2 ч Участники: PM, TL, дизайнер, PMO
3
О·0 → О·1 → Gate 1: первый полный виток
Команда проходит Орбиты 0 и 1 по фреймворку: бриф → скрининг PMO → Gate 0 → Разведка (custdev, синтез, Discovery Report, ADR) → Gate 1. PMO и CPO/CTO наблюдают, дают обратную связь на каждом Gate.
Длительность: 4–8 недель Чекпоинты: Gate 0 + Gate 1 Сопровождение: еженедельный синк с PMO
4
Ретро пилота + решение о масштабировании
После Gate 1: ретроспектива с пилотной командой (что работает, что надо адаптировать), презентация выводов CPO/CTO. Фиксируем изменения в фреймворк, запускаем следующие 2–3 инициативы.
Артефакт: Отчёт пилота Решение: масштабировать / доработать Следующий шаг: +2–3 инициативы
Критерии успеха
Когда пилот считается успешным
Процессные метрики
  • ✓ Бриф заполнен без повторных доработок (с 1–2 итерации)
  • ✓ Gate 0 пройден в срок (≤5 рабочих дней)
  • ✓ Разведка завершена в плановые сроки (±20%)
  • ✓ Discovery Report написан без помощи PMO (команда сама)
  • ✓ Gate 1 проведён в формате Gate Review (20 мин + Q&A)

  • ✓ Команда понимает разницу О·0 / О·1 / О·2 без объяснений
  • ✓ Стейкхолдер видит прозрачность процесса (нет вопросов "а где мы?")
  • ✓ PM знает какие артефакты нужны на каждой стадии
  • ✓ TL заполнил ADR самостоятельно
  • ✓ Шаблоны использованы (не созданы свои с нуля)
