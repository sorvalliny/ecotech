Принципы
Философия фреймворка
Шесть опорных принципов, которые определяют как мы думаем о продуктах и принимаем решения.
01
Ценность измерима
Каждая инициатива выходит на комитет с УТП: что даём пользователю, что получает RWB, как это измеряем. Без этого — не берём в роадмап.
02
Данные решают
Приоритет инициативы определяет ICE-балл: влияние на метрику, уверенность в гипотезе, сложность реализации. Чем больше подтверждений — данные, CustDev, конкурентный анализ — тем выше балл.
03
Скорость через SLA
Каждая стадия — Discovery, Delivery, рост — имеет таймбокс и чёткие условия перехода. Согласования сокращаем, приоритеты фиксируем на входе.
04
Бэклог живёт
Комитет меняет приоритеты — это норма. Инициативы не удаляем, а уточняем релизность в следующих итерациях. История решений сохраняется, технического долга не копим.
05
Прозрачность для комитета
Роадмап, прогнозы релизов и PnL по инициативам видны в любой момент. Комитет видит планы и ожидания поставки без запросов к команде.
06
Связь с целями
Каждая инициатива привязана к целям продукта и экосистемы. Нет связи с OKR и метриками — нет места в квартальном плане.

Слой Б
Полёт — цикл развития внутри орбиты
Как орбиты и полёт связаны

Орбита отвечает на вопрос: где мы сейчас?

Полёт отвечает на вопрос: как мы летим к следующей орбите?

Внутри каждой орбиты шаттл делает витки: Discovery → Delivery → Метрики. На Старте — один виток. В Growth — витки повторяются непрерывно. Когда витков достаточно и Точка манёвра пройдена — уходим на следующую орбиту.

Виток — один цикл Discovery → Delivery
В Growth витки повторяются для каждой инициативы. 3–7 параллельно.

Спонсоры и бизнес-лидеры
CEO, Исполнительный директор — стратегические инициативы. Руководители направлений — задачи из практики.
OKR квартала
Инициативы, вытекающие из целей продукта и компании на текущий период.
Данные и метрики
Аналитика показывает проблему — команда формирует инициативу с гипотезой.
Команда: техдолг и UX
Техдолг — планируется явно в 20% слоте спринта. UX-улучшения — по данным исследований.
⚠ Любая инициатива входит в Delivery только через Discovery-скрининг. Даже если задачу ставит CEO.
Планирование — от цели до задачи
Планирование: горизонты по стадиям
Инициатива (продуктовая или техническая) — центральное ядро. Всё, что ниже — декомпозиция инициативы.
Продуктовая инициатива
Создаёт ценность для пользователя или бизнеса. Обязательно: Problem statement + целевая метрика + гипотеза. Путь: Discovery → Gate 1 → Delivery → Метрика.
Техническая инициатива
Улучшает надёжность, скорость, DevEx или снижает риски. Обязательно: техрадар или risk log. Путь: TL → 20% слот спринта (без Gate).

Орбита 1 — Старт / Релиз минимального продукта
Один виток Discovery→Delivery · MVP в проде · без широкой аудитории
DISCOVERY + DELIVERY
Фокус команды
  • ·CustDev: валидируем Problem-Solution Fit
  • ·Tech spikes: оцениваем архитектурные риски
  • ·Строим минимальный scope: только то, что нужно для проверки
  • ·Первый деплой в прод с мониторингом и rollback
Цикл (один виток)
  1. CustDev + анализ рынка (2–3 нед)
  2. Техническое исследование, ADR, архитектурный набросок
  3. Концепция + бизнес-кейс → Gate 1 (Комитет)
  4. Delivery: спринты с чёткими DoD, feature flags
  5. MVP в прод → Gate 2 (Tech Lead + PM)
Правила
  • →Не обещать сроки и scope до конца Discovery
  • →Прототип ≠ продакшн-решение, не тащим в прод
  • →Все ключевые риски задокументированы в ADR
  • →MVP — не «плохо», MVP — «достаточно для проверки»
🛠 Стек инструментов
  • →Трекер (YouTrack / RWB Tracker)
  • →Figma (прототипы, wireframes)
  • →Wiki (PRD, ADR, CustDev-репорт)
  • →BI (базовые метрики MVP)
  • →CI/CD конвейер (первый деплой)
✅ Обязательные процессы
  • ★CustDev ≥ 5 интервью (до Delivery)
  • ★ADR: архитектурные решения
  • ★Бюджет согласован с Финансами
  • ★Архитектурный ревью (ИТ)
  • ★Runbook + мониторинг до Gate 2
  • ★Security review (ИБ)

Handbook · Орбита 1 · Discovery → MVP
PM — Discovery (нед. 1–4)
1. CustDev-скрипт: 5–7 интервью, HMW-синтез с DA
2. Определи North Star Metric и Activation-метрику
3. Прототип в Figma → usability тест (5 чел.)
4. Напиши PRD: scope, acceptance criteria, OKD
5. Защита на Gate 1 → Delivery
TL — Discovery (нед. 1–4)
1. Tech spike: прощупай архитектурные риски (2–3 дня)
2. Напиши ADR: выбранный подход и отклонённые альтернативы
3. Оцени интеграции экосистемы: WB ID, SDK Video, Корзина
4. Настрой CI/CD skeleton и feature flags с первого дня
5. Определи SLO для MVP: uptime, latency, error rate
PM — Delivery (нед. 5–12)
1. Веди Backlog: User Stories с чёткими DoD
2. Спринт-ревью с заказчиком каждые 2 нед.
3. Мониторь Activation Rate с первого дня в проде
4. Не расширяй scope без Gate
5. Готовь данные для Gate 2
TL — Delivery (нед. 5–12)
1. Code review 100% PR перед мержем в main
2. Runbook: деплой, откат, алерты — до выхода в прод
3. Security review с ИБ до Gate 2
4. DORA-метрики с первого спринта: deploy freq, MTTR
5. Постинцидентный анализ любого P0

PM — каждый виток роста
