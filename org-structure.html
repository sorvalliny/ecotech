Роль Проектного офиса в целевой структуре

Что делает Единая картина по всем проектам. Контроль сроков и инициатив. Приоритизация на данных. Отчётность CEO.
Чем отличается от CPO/CTO CPO — стратегия продуктов ("что делаем"). CTO — разработка ("как делаем"). PMO — координация ("делаем ли в срок и по приоритету").
Как взаимодействует PM продуктов функционально отчитываются в ПО (30 мин/нед). Направления — через владельцев. CEO получает статус без запросов.
Три сценария структуры
Общий вызов: вывести департамент на монетизацию и рост коммерческих показателей
А. Продуктовый
Каждый продукт — отдельный P&L
Product Lead-ы автономны: свой бюджет, команда, метрики. Быстрый вывод в прод. CPO — стратегия портфеля. ПО — координация и контроль.
+ Скорость монетизации зрелых продуктов
+ Ответственность за P&L у владельца
+ Конкуренция между продуктами за ресурс
− Дублирование функций (каждому свой маркетинг)
− Ранние продукты проигрывают за ресурсы
Подходит: когда 3+ продукта уже в проде и готовы к P&L
Б. Платформенный
Общая инфраструктура, единая монетизация
CTO строит единую платформу. Продукты — сервисы внутри экосистемы. Монетизация общая: подписка, маркетплейс, B2B. ПО — lifecycle и стандарты.
+ Экономия на инфраструктуре
+ Единый пользовательский опыт
+ Кросс-продуктовая монетизация
− Медленный старт (сначала платформа)
− Зависимость всех от CTO
Подходит: когда есть сильный CTO и продукты связаны
В. Гибридный рек.
Зрелые — автономный P&L, ранние — инкубатор
Wibes, Мост, Tingli — автономные P&L-юниты. Kids, Sport, Встречи, Книги — инкубатор с общими ресурсами (маркетинг, контент, аналитика). Направления — отдельный контур. ПО — сквозная координация.
+ Зрелые зарабатывают, не ждут
+ Ранние не конкурируют за ресурс со зрелыми
+ Направления не мешают ИТ-контуру
+ Общие функции экономят ФОТ
− Сложнее управлять (два режима)
− Нужен сильный ПО для координации
Подходит: текущая ситуация — разная зрелость продуктов
Процесс трансформации
Фаза 1: Апрель
Ничего не ломаем.
ПО встраивается как координатор поверх текущей структуры.
PM продолжают работать как раньше + 30 мин/нед отчётность.
Направления не трогаем.
→
Фаза 2: Май-Июнь
Вертикали формируются.
CPO/CTO/Монетизация/Аналитика получают явный контур.
ПО ведёт портфель ИТ-продуктов.
Направления пока при CEO.
→
Фаза 3: Q3+
Направления структурируются.
Отдельная модель для не-ИТ.
CEO видит 5-6 вертикалей вместо 14+ подчинений.
Полная прозрачность.
Как структура влияет на процессы
AS-IS: плоская
CEO — единая точка принятия решений по всему
Приоритеты конкурируют: продукт vs направление vs функция
Статус — только при личном запросе
Нет единого lifecycle — каждый по-своему
Кросс-проекты — через CEO лично
TO-BE: вертикали
CEO принимает стратегические решения, операционка — в вертикалях
Приоритеты — по RICE внутри контура, CEO решает между контурами
Статус — автоматически через ПО, без запроса
Единый lifecycle (Stage-Gate) для всех ИТ-продуктов
Кросс-проекты — через ПО, не через CEO
Принцип: ничего не ломаем. Сначала надстраиваем координацию, потом формализуем структуру.
Целевой подход — на обсуждение. Предлагаем начать с контура 1 (ИТ-продукты), расширить на контур 2 по результатам.
5 вертикалей агрегируют 14+ прямых подчинений в управляемую структуру.
