## БЛОК 1: ТРЕКЕР — ЯДРО ПЛАТФОРМЫ
*Трекер должен быть единым центром, а не одной из страниц*

### T-01 | Вкладка «Цели» открыта по умолчанию, а видно «Продукты»
- **Проблема:** UI баг — таб «Цели» активен, контент показывает «Продукты»
- **Усилие:** 0.5 дня
- **Критичность для пилота:** БЛОКЕР

### T-02 | Продукты не связаны с Целями
- **Проблема:** Цель (OKR) живёт отдельно от Продукта. Нет поля productId в целях, нет отображения целей на карточке продукта.
- **Что нужно:** Цель привязана к Продукту. На карточке продукта видны его цели и KR.
- **Усилие:** 2 дня
- **Критичность:** ВЫСОКАЯ

### T-03 | KR не связаны с Инициативами
- **Проблема:** Key Result — это измеримый результат. Инициатива — это работа для его достижения. Связи нет.
- **Что нужно:** KR → привязка к инициативам, которые на него работают. Прогресс KR = агрегат прогресса инициатив.
- **Усилие:** 3 дня
- **Критичность:** ВЫСОКАЯ

### T-04 | Нет таблицы прогресса KR по месяцам
- **Проблема:** KR имеет target/current, но нет истории. Нельзя увидеть тренд.
- **Что нужно:** Таблица: KR × месяцы, значение на конец каждого месяца, вклад в Objective.
- **Усилие:** 2 дня
- **Критичность:** СРЕДНЯЯ (для пилота можно вести в Excel)

### T-05 | Таймлайн фильтров в реестре не работает
- **Проблема:** UI баг — фильтр по дате/кварталу не функционирует.
- **Усилие:** 1 день
- **Критичность:** СРЕДНЯЯ

### T-06 | Статистику убрать в подвал
- **Проблема:** KPI-strip занимает верх экрана, а PM хочет сразу видеть продукты.
- **Что нужно:** Статистику свернуть или убрать в footer / collapsible.
- **Усилие:** 0.5 дня
- **Критичность:** НИЗКАЯ

### T-07 | Нужен переключатель: Витрина / Список / Канбан
- **Проблема:** Продукты только в карточном виде. Нет списка (таблицы) и Канбан-доски по стадиям.
- **Что нужно:** 3 view: Cards (текущий), Table (список), Kanban (колонки = Орбиты О·0–О·6).
- **Для инициатив:** Kanban по статусу (Бэклог → Запланировано → В работе → Готово).
- **Усилие:** 4 дня
- **Критичность:** ВЫСОКАЯ (Kanban — основной инструмент PM)

### T-08 | Эпики на карточке продукта → Инициативы
- **Проблема:** Карточка продукта показывает эпики, а должна показывать инициативы (сегментированные: бизнес / продукт / техника).
- **Что нужно:** Инициатива = основная единица. Может быть комплексной (несколько сегментов и команд).
- **Усилие:** 3 дня
- **Критичность:** ВЫСОКАЯ

### T-09 | Логи изменений (Changelog)
- **Проблема:** Нет истории кто, когда и что изменил в инициативе/продукте.
- **Что нужно:** При каждом сохранении — запись: дата, автор, что изменено. Лог на карточке.
- **Усилие:** 2 дня
- **Критичность:** ВЫСОКАЯ (для аудита и доверия)

### T-10 | Процесс валидации изменений
- **Проблема:** Любой может изменить что угодно. Нет workflow утверждения.
- **Что нужно:** Роли: Инициатор (предлагает изменение) → Апрувер (утверждает) → Наблюдатель (получает уведомление). Изменения в статусе ≥ wip требуют апрува.
- **Усилие:** 5 дней
- **Критичность:** ВЫСОКАЯ, но зависит от MVP авторизации (T-15)

## БЛОК 2: РОЛЕВАЯ МОДЕЛЬ И АВТОРИЗАЦИЯ
*Без ролей платформа = блокнот, а не система управления*

### R-01 | MVP авторизации (ID + роли)
- **Проблема:** Нет идентификации пользователя. Все видят и могут всё.
- **Что нужно (MVP):** Простой login (имя + роль из списка), сохранение в localStorage. Без пароля (внутренняя сеть). Роли: Admin, PMO, PM, Viewer.
- **Усилие:** 3 дня
- **Критичность:** БЛОКЕР для T-10, T-14

### R-02 | Матрица ролей и прав
- **Проблема:** Нет определения кто что может.
- **Что нужно:**

| Действие | Admin | PMO | PM (свой продукт) | PM (чужой) | Viewer |
|----------|-------|-----|-------------------|-----------|--------|
| Создать продукт | Да | Нет | Нет | Нет | Нет |
| Назначить владельца | Да | Нет | Нет | Нет | Нет |
| Создать цель | Да | Да | Да | Нет | Нет |
| Создать инициативу | Да | Да | Да | Нет | Нет |
| Изменить инициативу | Да | Да | Да | Нет | Нет |
| Изменить статус → wip | Да | Да | Да | Нет | Нет |
| Изменить статус → done | Да | Апрув | Запрос | Нет | Нет |
| Запросить Gate Review | Да | Да | Да | Нет | Нет |
| Принять Gate решение | Да | Да | Нет | Нет | Нет |
| Смотреть всё | Да | Да | Да | Да | Да |

- **Усилие:** 2 дня (дизайн + реализация)
- **Критичность:** ВЫСОКАЯ

### R-03 | Контактная база с ролями
- **Проблема:** Нет справочника «кто есть кто» с привязкой к ролям и ссылкой на MyTeam.
- **Что нужно:** Справочник: Имя, Роль (CEO/CPO/CTO/PMO/PM/TL), Продукт, Контакт (ссылка на MyTeam), Email.
- **Усилие:** 1 день
- **Критичность:** СРЕДНЯЯ

### R-04 | Нотификации по ролям
- **Проблема:** Никто не получает уведомлений о событиях.
- **Что нужно (MVP):** При изменении статуса/Gate — показать toast и записать в лог. Полные нотификации (email/messenger) — после пилота.
- **Усилие:** 1 день (toast + лог), 5 дней (email/messenger)
- **Критичность:** СРЕДНЯЯ для пилота (живые синки заменят)

## БЛОК 3: ЦЕЛИ И ПЛАНИРОВАНИЕ
*Годовое планирование CEO→PM через цели — реальный бизнес-кейс*

### P-01 | Процесс работы с целями не описан
- **Проблема:** Есть CRUD для целей, но нет описания: кто ставит, когда, как каскадируется, как декомпозируется в инициативы.
- **Что нужно:** Процесс: CEO ставит стратегические цели → CPO декомпозирует в продуктовые → PM создаёт инициативы для достижения KR.
- **Усилие:** 2 дня (описание + UI flow)
- **Критичность:** ВЫСОКАЯ

### P-02 | Фокус на Инициативу как основную единицу планирования
- **Проблема:** Сейчас трекер показывает продукты первыми. Но PM работает с инициативами.
- **Что нужно:** Инициатива = центр. Продукт = контекст. Цель = причина. Вкладка «Инициативы» = основная рабочая область PM.
- **Усилие:** 2 дня
- **Критичность:** ВЫСОКАЯ

### P-03 | Инициатива = комплексная единица
- **Проблема:** Инициатива сейчас плоская (name, desc, RICE). Нет сегментации.
- **Что нужно:** Инициатива содержит сегменты: Бизнес (монетизация), Продуктовый (фичи, UX), Технический (архитектура, инфра). Может затрагивать несколько команд.
- **Усилие:** 3 дня
- **Критичность:** СРЕДНЯЯ (для пилота можно в desc)

## БЛОК 4: БРИФ И ШАБЛОНЫ
*Шаблоны — живые инструменты, а не текст*

### S-01 | Product Brief = бриф нового Продукта (не инициативы)
- **Проблема:** brief.html создаёт инициативу в бэклоге. Но для бизнеса это заявка на новый продукт.
- **Что нужно:**
  - brief.html → «Заявка на новый продукт» (CTA на business.html)
  - В трекере: «Создать инициативу» (быстрая кнопка для PM)
  - В навигацию: быстрые кнопки «Создать цель», «Создать инициативу»
- **Усилие:** 2 дня
- **Критичность:** ВЫСОКАЯ

### S-02 | Бриф должен содержать подсказки по артефактам
- **Проблема:** Бриф = форма без контекста. PM не знает какие артефакты приложить.
- **Что нужно:** В брифе подсказки: «Приложите исследование», «Приложите концепцию», ссылки на шаблоны для каждого артефакта.
- **Усилие:** 1 день
- **Критичность:** СРЕДНЯЯ

### S-03 | Шаблоны → живые инструменты
- **Проблема:** templates.html = текстовые описания. Не заполняемые формы.
- **Что нужно:** Каждый шаблон = интерактивная форма (как brief.html). Заполнил → сохранил → привязал к инициативе.
- **Усилие:** 10 дней (12 шаблонов × ~1 день)
- **Критичность:** СРЕДНЯЯ (для пилота 2–3 ключевых: Brief, Gate Review Deck, Отчёт Разведки)

## БЛОК 5: GATE-ЧЕКЛИСТ И ФРЕЙМВОРК
*Связать чеклист с реестром, раскрыть виток*

### F-01 | Gate-чеклист не связан с реестром продуктов
- **Проблема:** Чеклист живёт в отдельном localStorage. Трекер не знает % готовности Gate.
- **Что нужно:** Прогресс Gate-чеклиста виден на карточке продукта в трекере.
- **Усилие:** 1 день
- **Критичность:** ВЫСОКАЯ

### F-02 | Framework-lite переименовать в «Как мы работаем по стадиям»
- **Проблема:** Название непонятно для команд.
- **Усилие:** 0.5 дня
- **Критичность:** БЛОКЕР (перед онбордингом)

### F-03 | В стадиях нет разделения Бизнес / Продукт / Разработка
- **Проблема:** На каждой стадии намешаны процессы, артефакты и шаблоны без разделения по направлениям.
- **Что нужно:** На каждой стадии 3 трека:
  - Бизнес: P&L, монетизация, GTM, юридические
  - Продукт: CustDev, PRD, UX, метрики
  - Разработка: ADR, архитектура, CI/CD, SLA
- **Усилие:** 5 дней
- **Критичность:** ВЫСОКАЯ

### F-04 | Виток (Исследование→Разработка→Оценка) не раскрыт
- **Проблема:** Упоминается поверхностно. Не описано как виток усложняется на разных стадиях.
- **Что нужно:** На О·0–О·1 виток простой (1–2 нед). На О·3–О·4 виток сложный (A/B, Growth, масштабирование). Описать для каждой стадии.
- **Усилие:** 3 дня
- **Критичность:** ВЫСОКАЯ

### F-05 | Вкладка «Пилот» устарела — убрать или доработать
- **Проблема:** Описывает старый процесс пилотного запуска.
- **Что нужно:** Убрать. Пилот = живой процесс, не вкладка.
- **Усилие:** 0.5 дня
- **Критичность:** БЛОКЕР

### F-06 | Роли (ответственные комитеты) на каждой стадии
- **Проблема:** На каждой стадии есть Gate-комитет, но нет контактов.
- **Что нужно:** На каждой стадии: кто ответственный, ссылка на контакт в MyTeam.
- **Усилие:** 1 день (после R-03)
- **Критичность:** СРЕДНЯЯ

## БЛОК 6: НАВИГАЦИЯ И ПОИСК
*Платформа должна вести, а не путать*

### N-01 | Поиск показывает страницы не из пилота
- **Проблема:** search-index.json содержит записи удалённых/замороженных страниц.
- **Что нужно:** Вычистить search-index.json — только пилотные страницы.
- **Усилие:** 0.5 дня
- **Критичность:** БЛОКЕР

### N-02 | Быстрые кнопки в навигации
- **Проблема:** Нет быстрого способа создать цель/инициативу без захода в трекер.
- **Что нужно:** В навигации или на product.html: «+ Создать цель», «+ Создать инициативу».
- **Усилие:** 1 день
- **Критичность:** СРЕДНЯЯ

### N-03 | Витрина продуктов → отдельный раздел в навигации
- **Проблема:** business.html показывает презентацию, а не живой портфель.
- **Что нужно:** Разделить:
  - «Витрина продуктов» = живые карточки из трекера (для CEO/CPO)
  - «Реестр инициатив» = мониторинг (для PMO)
  - «Трекер» = управление (для PM)
- **Усилие:** 3 дня
- **Критичность:** СРЕДНЯЯ

## БЛОК 7: MVP АДМИНКИ
*Без админки нет ролей, без ролей нет процесса*

### A-01 | MVP админки
- **Проблема:** Нет способа управлять пользователями, ролями, правами.
- **Что нужно (MVP):**
  - Страница /admin.html (доступна только Admin)
  - Список пользователей: имя, роль, продукт, email
  - Назначение ролей (Admin/PMO/PM/Viewer)
  - Создание продуктов (только Admin)
  - Данные в localStorage: ECOTECH_USERS, ECOTECH_AUTH
- **Усилие:** 5 дней
- **Критичность:** ВЫСОКАЯ

### A-02 | Убрать встроенные редакторы страниц
- **Проблема:** На некоторых страницах (фреймворк, бизнес) есть inline-редактор `data-editable`. Работает неконсистентно.
- **Что нужно:** Убрать все `data-editable`. Единый редактор = Admin Panel.
- **Усилие:** 1 день
- **Критичность:** СРЕДНЯЯ

### A-03 | Убрать нерабочий редактор стадий
- **Проблема:** В framework-lite есть кнопка редактирования стадий, но она не работает.
- **Что нужно:** Убрать до реализации полноценной админки.
- **Усилие:** 0.5 дня
- **Критичность:** БЛОКЕР

## БЛОК 8: ВИТРИНА ДЛЯ БИЗНЕСА
*Презентация ≠ инструмент*

### B-01 | Витрина не связана с данными трекера
- **Проблема:** business.html показывает презентацию и статичные слайды, а не живые данные.
- **Что нужно:** Секция портфеля на business.html = карточки из ECOTECH_PRODUCTS (уже частично работает через PT.ensureProducts, но слайды и орбитальная анимация — мусор для CEO).
- **Усилие:** 2 дня
- **Критичность:** СРЕДНЯЯ

### B-02 | CTA на business.html → подача заявки на продукт
- **Проблема:** CTA «Давайте создадим» ведёт на внешнюю форму.
- **Что нужно:** CTA → brief.html (подача заявки на новый продукт).
- **Усилие:** 0.5 дня
- **Критичность:** ВЫСОКАЯ

### B-03 | Презентация содержит «воду»
- **Проблема:** 10 слайдов с общими фразами. CEO хочет цифры, не философию.
- **Что нужно:** Упростить до 3–4 слайдов: портфель (живые данные), метрики, Gate-воронка, контакт.
- **Усилие:** 2 дня
- **Критичность:** СРЕДНЯЯ

## ВЕРДИКТ

Платформа требует **~4 дня работы на блокеры** перед запуском пилота и **~15 дней на критичные фичи** в первую неделю. Это реалистично при параллельной работе 2–3 агентов.

Рекомендация: **запустить пилот после блокеров (4 дня)**, критичные фичи дорабатывать параллельно с живым фидбеком от команд.
