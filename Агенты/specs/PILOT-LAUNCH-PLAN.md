# ПЛАН ЗАПУСКА ПИЛОТА ОРБИТА
## 16 марта 2026 | Методолог → Руководителю ПО

---

## ФОРМАТ ПИЛОТА

**Не платформа ведёт команды, а PMO ведёт команды через платформу.**

Платформа сейчас — инструмент PMO, а не self-service для команд. Это нормально для пилота. Self-service = после авторизации и shared storage.

---

## ПАРАМЕТРЫ

| Параметр | Значение |
|----------|---------|
| Длительность | 4–8 недель |
| Команды | 2–3 продукта на разных Орбитах |
| Ведущий | PMO (PMO Lead) |
| Данные | PMO вводит в трекер, команды видят на демо |
| Ритуалы | Установочная встреча + еженедельный синк |

---

## ПЕРЕД ЗАПУСКОМ (1–2 дня)

### 1. Аудит трекера (V-01)
Пройти каждый сценарий руками:
- Создать продукт → проверить карточку
- Создать цель → привязать к продукту → добавить KR
- Создать инициативу → привязать к цели и продукту
- Открыть Gate-чеклист → заполнить → проверить прогресс на карточке
- Переключить views (Cards/Table/Kanban)
- Зафиксировать все баги → исправить критичные

### 2. Подготовить данные для пилотных продуктов
В трекер внести:
- 2–3 реальных продукта экосистемы (название, описание, Орбита, владелец, trackerUrl)
- Для каждого: 1–2 цели с KR
- Для каждого: 3–5 инициатив с RICE-оценкой
- Привязать инициативы к KR

### 3. Подготовить Gate-чеклисты
Для каждого продукта открыть gate-checklist.html и заполнить текущий прогресс.

---

## НЕДЕЛЯ 0: УСТАНОВОЧНАЯ ВСТРЕЧА (2 часа)

### Участники
- PMO (ведущий)
- PM каждого пилотного продукта
- TL каждого продукта (опционально)

### Агенда

**Блок 1: Контекст (20 мин)**
- Зачем ОРБИТА, что решает
- Орбиты О·0–О·6: «Где ваш продукт сейчас?»
- Виток: Исследование → Разработка → Оценка и ретро

**Блок 2: Демо платформы (30 мин)**
- Трекер: карточка продукта, цели, инициативы
- «Как мы работаем»: стадия продукта → артефакты → Gate
- Gate-чеклист: что нужно для перехода
- Заявка на инициативу: brief.html
- Шаблоны: templates.html

**Блок 3: Практика (40 мин)**
- Каждый PM находит свой продукт в трекере
- Проверяет: правильная ли Орбита, цели, инициативы
- Открывает Gate-чеклист и оценивает готовность
- Создаёт 1 новую инициативу

**Блок 4: Договорённости (30 мин)**
- Еженедельный синк: день, время, формат
- Кто обновляет трекер (PMO на пилоте, PM после)
- Когда первый Gate Review
- Каналы коммуникации (Messenger/Band)
- Обратная связь: что неудобно, что непонятно → фиксируем

---

## НЕДЕЛИ 1–4: ЕЖЕНЕДЕЛЬНЫЙ СИНК (30 мин)

### Формат

**Для каждого продукта (10 мин):**
1. PM: Что сделали за неделю? Какие инициативы двигались?
2. PMO: Обновляет статус в трекере (при команде)
3. Вместе: Готовы к Gate? Что осталось по чеклисту?
4. Блокеры → кому эскалировать

**Общая часть (10 мин):**
5. PMO: Фидбек по платформе — что сломано, что непонятно
6. Планы на следующую неделю

### После синка
PMO обновляет трекер:
- Статусы инициатив
- Прогресс KR
- Gate-чеклист
- Лог изменений (пока вручную в desc)

---

## НЕДЕЛЯ 3–4: ПЕРВЫЙ GATE REVIEW

### Подготовка (за 3 дня)
1. PM проверяет Gate-чеклист → все обязательные = ✓
2. PM готовит артефакты по шаблонам (templates.html)
3. PMO проверяет готовность, назначает встречу
4. PMO отправляет приглашение комитету

### Проведение (40 мин)
По формату из AGENT_FACILITATOR.md:
- Контекст (5 мин) → Презентация (15 мин) → Вопросы (10 мин) → Решение (5 мин) → Follow-up (5 мин)

### После Gate Review
- PMO фиксирует решение (Go/No-Go) в трекере
- Обновляет Орбиту продукта если Go
- PM получает action items

---

## НЕДЕЛЯ 6–8: РЕТРО ПИЛОТА

### Вопросы для обсуждения
1. Что работало? Какие инструменты реально помогли?
2. Что не работало? Что вызвало путаницу?
3. Чего не хватало? Что пришлось делать вне платформы?
4. Терминология: команды говорят «Орбита» или по-прежнему «стадия»?
5. Процесс целеполагания: помог или формальность?
6. Gate Review: формат подошёл?

### Артефакт
Отчёт пилота → в бэклог ОРБИТА → корректировка платформы → волна 2

---

## КРИТЕРИИ УСПЕХА ПИЛОТА

| Критерий | Измерение |
|----------|-----------|
| 100% пилотных продуктов в трекере | Карточки заполнены, цели привязаны |
| ≥1 продукт прошёл Gate Review | Go или обоснованный No-Go |
| Время от заявки до Gate 0 ≤ 5 дней | SLA соблюдён |
| PM используют платформу без помощи PMO | К концу пилота — самостоятельно |
| Фидбек собран и занесён в бэклог | ≥10 конкретных улучшений |

---

## ВЫБОР ПИЛОТНЫХ ПРОДУКТОВ

### Идеальная комбинация:
- **Продукт на О·0–О·1** (ранняя стадия) — проверить процесс заявки, CustDev, Gate 0–1
- **Продукт на О·2–О·3** (активная разработка) — проверить трекинг инициатив, спринты, Gate 2–3
- **Продукт на О·4+** (зрелый) — проверить метрики, P&L, стратегический ревью

### Критерии выбора:
- Мотивированный PM (готов тратить 30 мин/нед на платформу)
- Продукт с реальной активностью (не замороженный)
- Разные категории (B2C + B2B или B2C + Internal)

---

## ЧЕГО НЕ ДЕЛАТЬ НА ПИЛОТЕ

1. **Не требовать от команд самостоятельно вводить данные** — PMO вводит при команде
2. **Не настаивать на полном заполнении всех полей** — минимум: название, Орбита, 1 цель, 2–3 инициативы
3. **Не проводить Gate Review формально** — лучше один настоящий, чем три «для галочки»
4. **Не менять платформу каждый день** — фиксировать фидбек, исправлять пачками раз в неделю
5. **Не масштабировать до окончания пилота** — 2–3 команды, не больше
