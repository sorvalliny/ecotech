## ТВОЯ ЗОНА ОТВЕТСТВЕННОСТИ

### 1. Корпоративные системы — карта интеграций

| Система | Назначение | Что синхронизируем | Направление |
|---------|-----------|-------------------|-------------|
| **Shiva** | Командное управление | Состав команд, роли, PM/TL | Shiva → ОРБИТА |
| **WB Tracker** | Портфельное управление (эпики) | Эпики, статусы, Gate-переходы | ОРБИТА ↔ WB Tracker |
| **YouTrack / Jira** | Задачи и спринты | Story, Task, Sprint Velocity | Трекер → ОРБИТА |
| **Confluence / WB Wiki** | Документация | Ссылки на артефакты, PRD | ОРБИТА → Wiki |
| **Amplitude / PostHog** | Продуктовая аналитика | Retention, Activation, DAU | Analytics → ОРБИТА |
| **Metabase / Looker** | BI и отчётность | Revenue, GMV, P&L | BI → ОРБИТА |
| **Slack / WB Messenger** | Коммуникации | Уведомления о Gate, блокеры | ОРБИТА → Messenger |
| **Notion** | Заметки и планирование | Брифы, OKR, roadmap | Notion ↔ ОРБИТА |

### 2. Сценарии интеграции

**Сценарий A: Gate-переход**
```
1. PM нажимает «Запросить Gate Review» в ОРБИТА
2. → Создаётся событие в WB Tracker (эпик со статусом Gate Review)
3. → Уведомление в Slack/Messenger: комитету + PMO
4. → Автоматически подтягиваются метрики из Amplitude
5. → После решения: статус обновляется в ОРБИТА + WB Tracker
```

**Сценарий B: Новая инициатива**
```
1. PM заполняет Product Brief в ОРБИТА (brief.html)
2. → Инициатива создаётся в ECOTECH_BACKLOG
3. → Карточка создаётся в WB Tracker
4. → Уведомление PMO в Messenger
5. → PMO проводит скрининг, решение возвращается в ОРБИТА
```

**Сценарий C: Метрики для Gate Review**
```
1. Перед Gate Review: ОРБИТА запрашивает метрики
2. → Amplitude: Retention D7/D30, Activation Rate, DAU
3. → Metabase: Revenue, CAC, LTV
4. → YouTrack: Sprint Velocity, Lead Time
5. → Данные агрегируются в Product Health Dashboard
```

**Сценарий D: Kill Signal**
```
1. Аналитик фиксирует Kill-условие в ОРБИТА
2. → Автоматическое уведомление PMO + CPO
3. → Создаётся эскалация в WB Tracker
4. → Блокируются новые эпики до решения
```

## ОГРАНИЧЕНИЯ ТЕКУЩЕЙ АРХИТЕКТУРЫ

| Ограничение | Влияние | План |
|-------------|---------|------|
| GitHub Pages = статика | Нет серверной логики, нет API | Интеграции через клиентский JS или внешние сервисы |
| localStorage = один браузер | Данные не шарятся между пользователями | Переход на backend / Notion API / Google Sheets API |
| Нет аутентификации | Нет ролей в ОРБИТА | Admin-режим через localStorage flag (временно) |
| Нет webhooks | Уведомления вручную | Telegram Bot / Slack Webhook как промежуточный слой |

### Промежуточные решения (до backend)
1. **Google Sheets как хранилище** — Apps Script API для синхронизации
2. **Notion API** — для брифов и документации
3. **Telegram Bot** — для уведомлений о Gate-переходах
4. **GitHub Actions** — для автоматизации data.json → деплой
