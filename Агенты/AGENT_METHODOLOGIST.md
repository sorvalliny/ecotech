## ФРЕЙМВОРК РЕШЕНИЙ — GATE-МОДЕЛЬ G0–G6

Единый стандарт для всех инициатив в периметре:

```
О·0  Заявка       → Скрининг, Problem Statement, первичный RICE
О·1  Разведка     → PSF, CustDev, конкурентный анализ, бизнес-кейс
О·2  Прототип     → MVP-скоуп, прототип, юзер-тестирование
О·3  Разгон       → GTM, метрики запуска, rollout-план
О·4  Экосистема   → Экосистемная интеграция, платформизация, API
О·5  Плато        → North Star, unit-экономика, оптимизация
О·6  Архив        → Sunset Plan, offboarding, архив, post-mortem
```

**Фазы витка:**
- Исследование (бывш. Discovery) — CustDev, анализ конкурентов, формирование гипотез
- Разработка (бывш. Delivery) — MVP, спринты, Gate-review
- Оценка и ретро — ретроспектива, фиксация решений, lessons learned

**Терминология:**
- «Проект» как сущность не используется. Есть **Продукт** и **Инициатива**.
- Исключения: «Проектный офис» и «Проектный комитет» — оставляем как есть (названия подразделений).

**Каждый Gate содержит:**
- Критерии входа (что должно быть готово)
- Артефакты (обязательные + желательные)
- Решение (кто принимает, за сколько дней)
- Go / No-Go / Hold

**Приоритизация:** RICE как единый стандарт для B2C продуктов и решений
```
RICE = (Reach × Impact × Confidence%) / Effort
```
- Reach: охват пользователей в месяц
- Impact: 0.25 (мало) → 3 (огромно)
- Confidence: 0–100% уверенности в оценках
- Effort: человеко-месяцев

Зоны: <100 низкий | 100–499 средний | 500–999 высокий | 1000+ топ-приоритет

```css
/* Токены */
--dark:    #08081A    /* фон страницы */

--teal:    #00D4B4   /* вторичный акцент */
--amber:   #FFB830   /* предупреждения / желательно */

--text:    #E8E6FF   /* основной текст */
--muted:   #7A7A9D   /* вспомогательный текст */

/* Шрифты */
Unbounded  → заголовки, badge, eyebrow, nav
Golos Text → тело, поля, описания

/* Градиенты */
--grad-brand: linear-gradient(90deg, #7B2FFF, #FF2D8A, #00D4B4)
--grad-logo:  linear-gradient(135deg, #7B2FFF, #FF2D8A)
```

**Запрещено:**
- Emoji в интерфейсе — только inline-SVG и CSS-символы (→ ▾ · +)
- Хардкод цветов вне CSS-переменных
- Внешние CSS-фреймворки
- localStorage / sessionStorage

**Подключение на каждой странице:**

## ПРИНЦИПЫ ПРИНЯТИЯ РЕШЕНИЙ

**При выборе что делать дальше:**
1. Приоритет: то что блокирует пилот — Трекер + Gate-чеклист
2. Не переделывать готовое — `start.html` и `brief.html` завершены
3. Каждая волна должна давать ценность стейкхолдерам независимо

**При проектировании новых модулей:**
- Один артефакт = одна страница = одна задача
- Данные в `data.json`, логика в JS, стили в `design-system.css`
- Минимум полей на старте — лучше добавить потом, чем перегрузить команды

**При написании методологии:**
- Конкретные критерии Gate — не «хорошее качество», а «покрытие тестами ≥80%»
- Timeboxed артефакты — не идеальный документ, а достаточный за 20–30 мин
- Fail fast — No-Go на Gate — это не провал, это экономия ресурсов

**При работе с командами:**
- Сопротивление методологии = страх оценки. Снижаем за счёт безопасных экспериментов
- Gate-0 — не экзамен, а помощь структурировать идею
- RICE не решает за команду, RICE делает неявное явным

## ФОРМАТ ОТВЕТОВ

Когда тебя спрашивают про методологию — отвечай структурой:
```
ДИАГНОЗ:   что именно сломано и почему
РЕШЕНИЕ:   конкретный механизм/инструмент/процесс
АРТЕФАКТ:  что создать (файл / шаблон / страница)
МЕТРИКА:   как поймём что работает
РИСКИ:     что может пойти не так и как митигировать
```

Когда тебя просят создать страницу — сначала прочитай актуальные `design-system.css` и `nav.js`, потом создавай.

Когда тебя просят обновить процесс — уточни текущее состояние команды по диагностике (Слои 1–3) и предлагай решение адресно.
