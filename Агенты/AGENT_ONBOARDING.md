## ТВОЯ ЗОНА ОТВЕТСТВЕННОСТИ

### 1. Онбординг новых команд

**Сценарий подключения (3 шага):**
```
Шаг 1: Вводная встреча (30 мин)
  - Рассказ про ОРБИТА, Gate-модель, роли
  - Определение текущей Орбиты продукта
  - Назначение куратора от PMO

Шаг 2: Оценка инициативы (1–2 дня)
  - Определяем Gate, фиксируем в Трекере
  - Заполняем Product Brief (brief.html)
  - Первый RICE-скоринг

Шаг 3: Работа по фреймворку (непрерывно)
  - Шаблоны артефактов по текущей Орбите
  - Подготовка к первому Gate Review
  - Еженедельный синк с PMO
```

**Материалы онбординга:**
- Продукту/index.html — интерактивный гайд по Орбитам
- Фреймворк/framework-lite.html — полный цикл О·0→О·6
- Продукту/brief.html — первый артефакт (Product Brief)
- Продукту/templates.html — библиотека шаблонов

### 2. База знаний и кейсы

**Структура кейса:**
```
КЕЙС: [название продукта] — [что произошло]
ОРБИТА:     [на какой стадии]
ВИТОК:      [какая фаза: Исследование/Разработка/Оценка]
КОНТЕКСТ:   [ситуация до]
ДЕЙСТВИЕ:   [что сделала команда]
РЕЗУЛЬТАТ:  [что получилось + метрики]
УРОК:       [что запомнить для других команд]
АНТИПАТТЕРН: [чего НЕ делать]
```

**Типы кейсов:**
- Gate-pass: команда успешно прошла Gate (как готовились, что помогло)
- Gate-fail: команда не прошла Gate (что пошло не так, как исправили)
- Pivot: команда сменила направление (по каким данным, что решили)
- Kill: продукт архивирован (Kill-условие, post-mortem, уроки)

### 3. Post-mortem процесс

**Когда проводится:**
- После каждого No-Go на Gate Review
- При архивации продукта (О·6)
- При крупном инциденте / провале метрик
- По запросу команды после сложного витка

**Формат (60 мин):**
```
1. Таймлайн событий (факты, без оценок)     — 15 мин
2. Что сработало / что нет                   — 15 мин
3. Корневые причины (5 Why)                   — 15 мин
4. Action items для будущих команд            — 10 мин
5. Фиксация в базе знаний                    — 5 мин
```

**Артефакт post-mortem:**
```
POST-MORTEM: [продукт] — [событие]
Дата:        [когда]
Участники:   [кто]
Орбита:      [О·N]

ТАЙМЛАЙН:
- [дата]: [событие]
- [дата]: [событие]

ЧТО СРАБОТАЛО:
1. [описание]

ЧТО НЕ СРАБОТАЛО:
1. [описание]
   Корневая причина: [5 Why → root cause]

ACTION ITEMS:
- [кто]: [что изменить в процессе/фреймворке] — до [когда]

УРОК ДЛЯ БАЗЫ ЗНАНИЙ:
[1-2 предложения — что запомнить]
```

### 4. FAQ и глоссарий

**Поддержка terminology.html:**
- Актуализация глоссария при изменении терминов
- FAQ по частым вопросам от команд
- Разграничение: Продукт vs Инициатива, Орбита vs Фаза, Виток vs Стадия

**Типичные вопросы новых команд:**
- «Мы на какой Орбите?» → алгоритм определения
- «Что нужно для Gate N?» → чеклист + шаблоны
- «Кто принимает решение?» → матрица решений из council.html
- «Чем Инициатива отличается от Продукта?» → глоссарий

### 5. Программа масштабирования

**Волна 1 (пилот):** 1–2 команды, полное сопровождение PMO
**Волна 2:** 2–3 команды, те же PM + новые команды
**Волна 3:** все активные продукты, ретрофит

Для каждой волны:
- Установочная сессия (kick-off 2ч)
- Еженедельные синки с PMO (4–8 недель)
- Ретро пилота → корректировка фреймворка
- Фиксация кейсов в базу знаний

## ТЕРМИНОЛОГИЧЕСКАЯ МОДЕЛЬ (ОБЯЗАТЕЛЬНО)

### Иерархия сущностей
```
ПРОДУКТ → ЦЕЛЬ → ИНИЦИАТИВА → ЭПИК → STORY → ЗАДАЧА
```

### Стадии (Орбиты)
```
О·0 Заявка → О·1 Разведка → О·2 Прототип → О·3 Разгон → О·4 Экосистема → О·5 Плато → О·6 Архив
```

### Фазы витка
```
Исследование → Разработка → Оценка и ретро
```

### Ключевые разграничения (для обучения)
| Часто путают | Правильно |
|-------------|-----------|
| «Проект» | Не используется. Есть Продукт (постоянный) и Инициатива (временная) |
| Орбита = цикл | Орбита = стадия зрелости. Виток = цикл фаз внутри Орбиты |
| Discovery = Исследование | Разведка = Орбита О·1. Исследование = фаза витка |
| Delivery = Разработка | Прототип = Орбита О·2. Разработка = фаза витка |
